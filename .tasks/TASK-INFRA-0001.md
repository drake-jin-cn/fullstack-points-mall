### Background

All 8 service repos currently contain only a `LICENSE` and `README.md`. No runnable application
skeleton exists in any of them. Before any feature work (auth, points, shop, etc.) can begin,
each service must have its framework-specific boilerplate in place so that:

1. The service starts cleanly with no errors.
2. A `GET /health` endpoint returns `200 OK` — the minimal signal that the process is up.
3. CI can verify liveness with a single HTTP check.

This task is the absolute prerequisite for every subsequent task in Phase 0 and beyond.

### Goals

1. `points-mall-bff` — NestJS skeleton with `GET /health → 200 OK`
2. `points-mall-core` — Spring Boot skeleton with `GET /health → 200 OK`
3. `points-mall-shop` — Laravel skeleton with `GET /health → 200 OK`
4. `points-mall-message` — Express + TypeScript skeleton with `GET /health → 200 OK`
5. `points-mall-thirdparty-connector` — Spring Boot WebFlux skeleton with `GET /health → 200 OK`
6. `points-mall-data` — FastAPI skeleton with `GET /health → 200 OK`
7. `points-mall-frontend` — Next.js (App Router) skeleton with API route `GET /api/health → 200 OK`
8. `points-mall-frontend-base` — Rollup + React component library skeleton that builds without error (`pnpm build` exits 0)

### Out of Scope

- No database connections (deferred to T003)
- No authentication / JWT (deferred to T006–T007)
- Multi-environment config and `.env.example` (deferred to T002)
- Docker Compose / container orchestration (deferred to later Phase 0 tasks)
- Enhanced health payload with `db`, `uptime`, `timestamp` fields (deferred to T004)
- Any business logic routes beyond `/health`

### Technical Design

#### Health response contract (all backend services)

```json
{ "status": "ok" }
```

HTTP `200 OK`, `Content-Type: application/json`.

**NestJS (bff)**
- Scaffold with `nest new` (pnpm workspace).
- Add `HealthModule` + `HealthController` returning `{ status: 'ok' }`.
- No database providers wired yet.

**Spring Boot (core)**
- Scaffold via Spring Initializr (Maven, spring-boot-starter-web, Java 21).
- Add `HealthController` with `@GetMapping("/health")` returning `ResponseEntity.ok(Map.of("status","ok"))`.
- Disable Spring Boot Actuator's own `/actuator/health` — use the custom route only.

**Laravel (shop)**
- Scaffold with `composer create-project laravel/laravel`.
- Register `Route::get('/health', fn() => response()->json(['status' => 'ok']))` in `routes/api.php`.
- No database migrations run at this stage.

**Express/TS (message)**
- `pnpm init`, install `express`, `typescript`, `@types/express`, `ts-node`, `tsup`.
- `src/index.ts`: create app, register `GET /health`, start listening.
- `tsconfig.json` targeting ES2022, `moduleResolution: node`.
- `pnpm dev` runs via `ts-node`; `pnpm build` compiles to `dist/` via `tsup`.

**Spring Boot WebFlux (thirdparty)**
- Maven project with `spring-boot-starter-webflux` (Netty server + WebClient).
- Controller returns `Mono<Map<String, String>>` — WebFlux 响应式返回类型。
- `mvn spring-boot:run` starts on port 8082.

**FastAPI (data)**
- `pyproject.toml` (or `requirements.txt`) with `fastapi` and `uvicorn[standard]`.
- `main.py`: `@app.get("/health")` returns `{"status": "ok"}`.
- `python -m uvicorn main:app` starts without error.

**Next.js (frontend)**
- Scaffold with `create-next-app` (App Router, TypeScript, Tailwind, ESLint).
- Add `app/api/health/route.ts` exporting `GET` handler returning `NextResponse.json({ status: 'ok' })`.

**Rollup/React (frontend-base)**
- `pnpm init`, install `react`, `react-dom`, `rollup`, `@rollup/plugin-typescript`, `rollup-plugin-peer-deps-external`.
- `rollup.config.ts` with ESM + CJS dual output.
- Export a single placeholder `Button` component so the bundle is non-empty.
- `pnpm build` exits 0.
