## Raw Requirements

`[core]` Employee credential verification API: `POST /internal/auth/verify` — validate email +
bcrypt password, return employee info. Endpoint is internal-only (requires `INTERNAL_API_KEY`
header, rejects all other callers). Employee seeder: one admin + two employee accounts. Database:
add `V7__add_password_hash_to_employees.sql` Flyway migration to add `password_hash VARCHAR NOT NULL`
column to existing `employees` table (created in Phase 0 without this field).

### Background

Phase 0 created the `employees` table (V2) with `password_hash VARCHAR(255)` nullable. Phase 1
requires Core to expose an internal-only credential verification endpoint so BFF can authenticate
employees via email + bcrypt before issuing JWTs.

### Goals

1. Add `POST /internal/auth/verify` — validate email + bcrypt password, return employee info.
2. Restrict endpoint to internal callers via `INTERNAL_API_KEY` header (no default value — startup fails if unset).
3. Enforce `password_hash NOT NULL` via Flyway migration V7.
4. Seed 1 admin + 2 employee accounts in `dev`/`test` environments only (`@Profile`).
5. Establish system-wide error code convention: service-prefixed enum (`CoreErrorCode`).

### Out of Scope

- JWT issuance (T007, BFF).
- GitHub OAuth / OIDC (T012–T014, T097–T099).
- Role/permission management beyond seeding existing roles.

#### API Contract

```
POST /internal/auth/verify
Header: INTERNAL_API_KEY: <value>
Body:   { "email": "...", "password": "..." }
```

Success `200`:
```json
{ "code": "OK", "message": "success", "data": { "id": 1, "name": "Admin", "email": "...", "isActive": true, "roles": ["admin"] } }
```

Error (example):
```json
{ "code": "core-1001", "message": "Invalid credentials", "data": null, "traceId": "<uuid>" }
```

#### Error Code Convention (system-wide)

| Service | Prefix | Range |
|---------|--------|-------|
| core | `core` | `core-1xxx` |
| bff | `bff` | `bff-2xxx` |
| thirdparty-connector | `tpc` | `tpc-3xxx` |
| shop | `shop` | `shop-4xxx` |
| message | `msg` | `msg-5xxx` |
| data | `data` | `data-6xxx` |

Propagation rule: BFF passes downstream `code` values through to frontend unchanged.
All exceptions mapped through `CoreErrorCode` enum — raw exceptions never reach the response.

#### CoreErrorCode Enum

| Code | HTTP | Scenario |
|------|------|----------|
| `core-1001` | 401 | Invalid credentials (email not found OR password mismatch — prevents enumeration) |
| `core-1002` | 403 | Account disabled |
| `core-1003` | 401 | Missing or invalid INTERNAL_API_KEY |
| `core-1010` | 400 | Request validation failed |
| `core-1099` | 500 | Unexpected internal error |

#### Response Envelope

```
// Success
{ "code": "OK", "message": "success", "data": <T> }

// Error (traceId on errors only)
{ "code": "<error-code>", "message": "<text>", "data": null, "traceId": "<uuid>" }
```

`traceId`: Core generates UUID per request (MDC). See sub-task below.

#### Employee Seeder (`@Profile({"dev","test"})`)

| Email | Password | Role |
|-------|----------|------|
| `admin@points-mall.com` | `Admin@123` | admin |
| `alice@points-mall.com` | `Employee@123` | employee |
| `bob@points-mall.com` | `Employee@123` | employee |

BCrypt strength: 12 rounds. Idempotent (skip if email already exists).

### Employee Seeder

- [x] AC-04 Seeder runs only under `dev` / `test` profiles; does not execute in `prod`
- [x] AC-05 After first startup, three employee accounts exist: `admin@pointsmall.com` (role: admin), `alice@pointsmall.com` (role: employee), `bob@pointsmall.com` (role: employee)
- [x] AC-06 Seeder is idempotent: repeated startups do not insert duplicates and do not throw errors
- [x] AC-07 All three accounts have `password_hash` encrypted with BCrypt 12 rounds, verifiable via BCrypt

### INTERNAL_API_KEY Interception (Filter)

- [x] AC-08 Requests to `/internal/**` without `INTERNAL_API_KEY` header return HTTP 401, body: `{ "code": "core-1003", "message": "Missing or invalid API key", "data": null, "traceId": "<uuid>" }`
- [x] AC-09 Requests with mismatched `INTERNAL_API_KEY` header value return HTTP 401, same body as AC-08
- [x] AC-10 Filter only intercepts `/internal/**` paths; other routes such as `/health` are unaffected

### POST /internal/auth/verify — Happy Path

- [x] AC-11 Valid API Key + valid email + valid password returns HTTP 200, body:
  ```json
  { "code": "OK", "message": "success", "data": { "id": <number>, "name": "<string>", "email": "<string>", "isActive": true, "roles": ["admin"] } }
  ```
- [x] AC-12 All three accounts (admin, alice, bob) can be verified and return their respective role lists

### POST /internal/auth/verify — Error Cases

- [x] AC-13 Non-existent email returns HTTP 401, `code: "core-1001"`, message **identical** to the wrong-password scenario (prevents user enumeration)
- [x] AC-14 Valid email but wrong password returns HTTP 401, `code: "core-1001"`, message **identical** to the non-existent email scenario
- [x] AC-15 Account with `is_active = false` returns HTTP 403, `code: "core-1002"`, message: `"Account disabled"`
- [x] AC-16 Request body missing `email` or `password` returns HTTP 400, `code: "core-1010"`, message: `"Request validation failed"`
- [x] AC-17 All error responses include a non-empty `traceId` field (UUID format); success responses do **not** contain `traceId`

### Security Constraints

- [x] AC-18 Response body exposes no internal error details (e.g. SQL exceptions, stack traces); unexpected errors uniformly return `core-1099`
- [x] AC-19 The `password` field must not appear in any log output

### Error Code Convention (System-wide)

- [x] AC-20 `CoreErrorCode` enum contains exactly five entries: `INVALID_CREDENTIALS(core-1001)`, `ACCOUNT_DISABLED(core-1002)`, `UNAUTHORIZED_CALLER(core-1003)`, `VALIDATION_FAILED(core-1010)`, `INTERNAL_ERROR(core-1099)`
- [x] AC-21 `GlobalExceptionHandler` catches all unhandled exceptions and downgrades them to `core-1099`; original exceptions are never re-thrown
